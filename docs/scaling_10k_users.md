# 支撑 1 万+ 同时在线：架构与配置

目标：**至少 1 万人同时使用聊天/实时功能无压力，并预留更高规模**。

---

## 1. Supabase Realtime 并发连接数（官方）

| 方案 | 并发连接数 | 适用 |
|------|-------------|------|
| Free | 200 | 开发/内测 |
| Pro | 500 | 小规模上线 |
| **Pro (no spend cap) / Team / Enterprise** | **10,000** | **1 万人在线** |
| Enterprise（定制） | 10,000+ | 2 万、5 万等更高 |

- 消息/秒、Channel 加入/秒：Pro no cap 及以上约 2,500/秒，一般够用。
- 需要 **1 万人在线**：至少选用 **Pro (no spend cap)** 或 **Team** 或 **Enterprise**。
- 需要 **超过 1 万连接**：联系 Supabase 开通 Enterprise 定制配额。

---

## 2. 控制「每用户连接数」才能撑满 1 万人

Realtime 按 **连接数** 计费/限流，不是按「注册用户数」。同一用户若订阅很多 channel，会占多个连接。

**建议在客户端做的优化（已做或待做）：**

- 只订阅 **当前需要的 channel**：例如只订阅「当前打开的会话」的消息流，离开会话就取消订阅。
- 会话列表：用 **一个** Realtime 流（如 `chat_members` 当前用户）驱动刷新，不要按每个会话各开一个 channel。
- 避免全局、长期订阅大量表或 channel；按页按需订阅，用完即关。

这样大致可以做到 **1 个在线用户 ≈ 1～2 个并发连接**。  
在「1 用户 ≈ 1 连接」的乐观估计下，**10,000 连接 ≈ 1 万同时在线**；若 1 用户 ≈ 2 连接，则 10,000 连接 ≈ 5,000 同时在线，要撑满 1 万人就需要 **20,000 连接**，需 Enterprise 定制。

---

## 3. 数据库与 API

- **PostgreSQL（Supabase）**：读多写少场景下，1 万 DAU/数千同时在线很常见；配合索引、RLS、连接池即可。
- 若后续有「热点会话」（单群几万人）：可考虑读从库、缓存热点会话元数据，避免单表单行过热。

---

## 4. 语音/视频通话（若接入）

- 使用 **Agora / 腾讯云 TRTC 等**：由厂商做全球扩容，单应用 **数万～数十万** 并发通话路数都可支持，按量计费。
- 通话与「聊天在线人数」独立：  
  - 聊天在线人数由 **Supabase Realtime 连接数** 决定；  
  - 同时进行音视频的人数由 **音视频厂商** 的并发与套餐决定。

---

## 5. 总结：如何保证 1 万+ 同时使用

| 项目 | 建议 |
|------|------|
| Supabase 方案 | 选 **Pro (no spend cap)** 或 **Team** 或 **Enterprise**，确保 Realtime 至少 10,000 连接。 |
| 客户端 | 按需订阅、只保留当前会话的 Realtime，控制每用户 1～2 连接。 |
| 更高规模（如 2 万、5 万在线） | 联系 Supabase 开 **Enterprise 定制** 提高 Realtime 连接上限。 |
| 音视频 | 用 Agora 等，可支持数万以上并发通话，与聊天在线人数解耦。 |

按上述配置，**最少保证 1 万人同时使用没有问题**；若要更高，提前谈好 Enterprise 与音视频套餐即可。

---

## 6. 后续升级清单（付费即可提升配额）

- **Supabase**  
  - 在 [Supabase Dashboard](https://supabase.com/dashboard) 里把项目从 Free/Pro 升级为 **Pro (no spend cap)** 或 **Team** 或 **Enterprise**。  
  - **升级后 Realtime 连接数会自动变为 10,000**，无需改代码。  
  - 需要超过 1 万连接时：选 Enterprise 并联系 Supabase 支持开通更高配额。

- **音视频（若接入 Agora 等）**  
  - 按用量/并发在厂商控制台购买套餐或按量计费即可，无需改应用逻辑。

**结论：扩容主要是「付费升级套餐」；代码侧只需保持「按需订阅、少占连接」的用法即可。**
