# Teacher Hub 代码体验报告（2026-03-17）

## 1. 审查范围与方法

- 覆盖模块：`tongxin-backend`、`tongxin-frontend`、`tongxin-admin` 的核心改动链路（鉴权、行情、交易员排行榜、后台配置管理）。
- 重点文件：管理员会话与登录、排行榜内容配置、交易员指标聚合、行情 API 代理、后台活动/卡片管理页面。
- 方法：静态代码审查 + 关键路径逻辑推演（未包含完整联调压测与真机 UI 回归）。

---

## 2. 体验结论（先看）

- 当前版本在「后台可配置化」与「行情兜底」方面有明显进步，整体方向正确。
- 但存在 **2 个高优先级逻辑问题**，会直接影响用户感知和榜单可信度。
- 管理端与业务命名出现偏差，后续维护成本会显著上升，建议尽快统一概念模型。

---

## 3. 主要问题清单（按严重级别）

### P0（高优先级，建议立即修复）

1. **排行榜切换榜单后未按对应维度重排**
   - 位置：`tongxin-frontend/lib/features/rankings/rankings_page.dart`
   - 现象：切换周榜/月榜/季榜/年榜时，仅显示分数字段变化，但列表顺序仍沿用原顺序（通常是总榜顺序）。
   - 影响：用户会认为“榜单切换无效”或“数据不可信”，这是核心体验风险。
   - 改进：在前端按当前 `_boardType` 对数据进行排序后再 `take(10)`，并补充切榜单用例测试。

2. **交易员 `pnl_current` 被映射为周已实现盈亏**
   - 位置：`tongxin-backend/lib/apiTeachers.js`
   - 现象：`applyTeacherPnlMetrics` 中 `pnl_current` 使用 `week_realized_pnl`，而非 `floating_pnl`。
   - 影响：页面“当前盈亏/浮盈浮亏”语义与真实数据不一致，可能导致交易判断误导。
   - 改进：将 `pnl_current` 映射到 `floating_pnl`，并核对前端字段文案（当前/本周）与接口语义一致。

### P1（中优先级，建议本迭代修复）

1. **排行榜内容轮询频率过高**
   - 位置：`tongxin-frontend/lib/features/rankings/rankings_page.dart`
   - 现象：页面每 5 秒请求一次 `api/rankings/content`。
   - 影响：移动端耗电、弱网抖动、后端无效请求堆积。
   - 改进：改为进入页面拉取 + 前后台切换时刷新；或 2~5 分钟轮询并加 ETag/版本号。

2. **“活动管理”命名与实际能力不一致**
   - 位置：`tongxin-admin/lib/pages/admin_activities_panel.dart` 与后端 `apiAdminActivities.js`/`apiAdminRankingsContent.js`
   - 现象：前端“活动管理”页面实际在管理“排行榜轮播卡”；而后端另有真正活动 CRUD 接口。
   - 影响：运营误操作、研发沟通成本升高、后续联调易错。
   - 改进：分离为“排行榜内容管理”和“活动管理”两个独立入口，路由与 API 命名对齐。

3. **上传举报截图回退到公共头像桶存在数据暴露风险**
   - 位置：`tongxin-backend/lib/apiUpload.js`
   - 现象：`report_screenshots` 不可用时回退到 `avatars` 并返回 public URL。
   - 影响：投诉材料可能暴露到非预期公开路径，存在合规风险。
   - 改进：改为“专用私有桶 + 签名 URL”，失败时明确返回错误，不应回退到公共桶。

### P2（优化项，可排期）

1. **后台会话 token 未持久化**
   - 位置：`tongxin-admin/lib/core/admin_api_client.dart`
   - 影响：刷新/重启后需重复登录，降低后台运营效率。
   - 改进：本地安全存储（如 `flutter_secure_storage`）并加过期校验。

2. **基础限流器内存增长风险**
   - 位置：`tongxin-backend/server.js`
   - 现象：内存 Map 仅累计不清理。
   - 影响：长时间运行在高来源 IP 场景会持续膨胀。
   - 改进：增加定时清理或切换成熟限流中间件（Redis/滑窗）。

---

## 4. 改进逻辑（落地顺序）

### 阶段 A：先修“信任问题”（1-2 天）

- 修复排行榜切榜排序逻辑。
- 修复 `pnl_current` 映射错误。
- 增加 3 组回归：榜单切换、交易员详情盈亏字段、排行榜前 10 结果一致性。

### 阶段 B：再修“稳定性与成本”（2-3 天）

- 将排行榜内容轮询从 5 秒改为低频或事件驱动刷新。
- 给后台配置接口增加轻量版本号，减少无效数据拉取。
- 优化接口错误提示（区分网络、鉴权、业务错误）。

### 阶段 C：最后做“架构与安全收敛”（3-5 天）

- 统一后台“活动/卡片”概念与菜单信息架构。
- 举报截图改为私有存储+签名访问。
- 替换/增强内存限流策略与观测指标。

---

## 5. 建议的验收标准（可直接作为测试单）

- 榜单切换后，前 10 名与所选维度排序严格一致。
- `pnl_current` 与数据库/函数中的浮动盈亏一致，不再与周收益混用。
- 排行榜内容接口平均请求频率下降（目标：单用户会话降低 70%+）。
- 后台“活动管理”与“排行榜卡片管理”页面职责清晰，运营可独立使用。
- 举报截图默认不可被公开 URL 直接遍历访问。

---

## 6. 总体评价

- **当前版本可用，但不建议直接放量运营。**
- 先完成 P0 后即可进入小范围灰度；P1 完成后再做更大规模推广。
- 本轮最关键不是“加功能”，而是先修正 **排行榜可信度** 与 **盈亏语义一致性**。
